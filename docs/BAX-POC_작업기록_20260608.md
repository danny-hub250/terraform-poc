## 1. 개요

본 문서는 BAX-POC 프로젝트의 Azure 인프라를 Terraform으로 검토·배포하고, 배포된 Linux VM에 접속하여 운영에 필요한 유틸리티(Azure CLI, kubectl)를 설치 및 점검한 전체 과정을 정리한 작업 기록입니다.

- 작업 범위: Terraform 코드 구조 분석 → Azure 배포 실행 → VM 접속 → 운영 도구 설치/점검
- 대상 환경: `environments/BaxPOC-DEV` (azurerm provider 4.63.0, Terraform >= 1.6)
- 대상 구독 ID: `d6ec08db-4ce1-4ed6-b16c-b4ff61504b4d`

## 2. Terraform 코드 구조 검토

### 2.1 모듈 구성

`modules/` 하위에 리소스 그룹, 네트워크(VNet/Subnet/NSG/Route Table/NAT Gateway/DNS), 컴퓨팅(Linux VM, Windows VM, AKS, Container Registry), AI(Foundry, OpenAI, AI Search), 데이터(PostgreSQL), 프라이빗 엔드포인트 등 총 18개 재사용 모듈이 정의되어 있으며, `environments/BaxPOC-DEV/main.tf`에서 이를 조합해 실제 배포 구성을 정의합니다.

### 2.2 BaxPOC-DEV 환경에서 호출된 주요 모듈

| 구분 | 리소스명 | 비고 |
|---|---|---|
| 리소스 그룹 | bax-poc-dev-app-rg / network-rg / ai-rg | 용도별 3개 분리 |
| 네트워크 | bax-poc-dev-vnet (10.130.0.0/24) | mgmt/pe/db/aks 4개 서브넷 |
| Linux VM | bax-poc-d-vm (Standard_D4s_v5) | 관리/점프박스 용도, Public IP 활성화 |
| AKS | bax-poc-dev-aks | system(sysnp01) + user(usernp01) 노드풀, `private_cluster_enabled=false` |
| Container Registry | baxpocdcr (Standard) | admin 활성화, 퍼블릭 액세스 허용 |
| AI Foundry | bax-poc-dev-msf (kind=AIServices) | Private Endpoint + 3개 Private DNS Zone 연결 |
| PostgreSQL | bax-poc-dev-psql (Flexible Server, PG 18) | Private Endpoint 구성 |

### 2.3 검토 중 식별된 주요 사항

- `provider "azurerm"`에 `subscription_id`가 명시되어 있지 않아 Azure CLI 로그인 컨텍스트의 구독으로 배포됨
- foundry 모듈의 `location`이 `"EastUS2"`로 하드코딩되어 있어 다른 리소스(`var.location=koreacentral`)와 리전이 상이함 — 의도 여부 확인 필요
- VM(`admin_password`) 및 PostgreSQL(`administrator_password`) 비밀번호가 `main.tf`에 평문으로 하드코딩되어 있음 → GitHub 커밋 전 `sensitive` 변수로 분리 처리
- `backend` 설정이 없어 상태 파일(`terraform.tfstate`)이 로컬에 저장됨 — 협업 시 원격 백엔드(Azure Storage Account 등) 구성 검토 필요
- `lifecycle`, `random`/`timestamp` 등 매 실행마다 diff를 유발할 수 있는 요소는 발견되지 않음 — 기본적으로 변경분만 반영되는 구조

## 3. Azure 배포 실행

### 3.1 사전 준비 사항

- Azure 구독 ID 확보 및 해당 구독에 대한 Contributor 이상 권한
- Azure CLI 설치 및 로그인 (provider가 별도 인증 정보 없이 `features {}`만 정의 → CLI 컨텍스트에 의존)
- Terraform 1.6 이상 설치
- `custom_subdomain_name`(예: `bax-poc-dev-msf`) 등 전역 고유 이름의 중복 여부 사전 확인
- 리전별 리소스 할당량(Quota) 및 `Microsoft.CognitiveServices`, `Microsoft.ContainerService` 등 Resource Provider 등록 여부 확인

### 3.2 실행 명령어 (`Azure/environments/BaxPOC-DEV` 디렉터리 기준)

Terraform은 현재 디렉터리의 `.tf` 파일을 루트 모듈로 인식하고, `main.tf`에서 모듈을 상대경로(`../../modules/...`)로 참조하므로 반드시 `Azure/environments/BaxPOC-DEV` 폴더에서 실행해야 합니다.

```bash
# 1) Azure 로그인 및 구독 전환
az login
az account set --subscription "d6ec08db-4ce1-4ed6-b16c-b4ff61504b4d"
az account show   # 대상 구독 ID로 전환되었는지 확인

# 2) 작업 디렉터리 이동
cd Azure/environments/BaxPOC-DEV

# 3) Terraform 초기화 / 검증
terraform init
terraform validate

# 4) 실행 계획 확인 후 적용
terraform plan -out=tfplan
terraform apply tfplan
```

### 3.3 트러블슈팅 — terraform init "empty directory"

최초 실행 시 저장소 루트(`bax-poc`)에서 `terraform init`을 수행하여 아래 메시지가 출력됨.

```
Terraform initialized in an empty directory!
```

이는 에러가 아니라 해당 위치에 `.tf` 파일이 없다는 안내로, `Azure/environments/BaxPOC-DEV`로 이동 후 재실행하여 정상 초기화함.

### 3.4 배포 결과

- `Azure/environments/BaxPOC-DEV/main.tf`에 정의된 전체 리소스(리소스 그룹 3종, VNet/Subnet, Linux VM, AKS, ACR, AI Foundry+PE, PostgreSQL+PE, Private DNS Zone 등) 배포 완료
- 배포 완료 후 `terraform.tfstate`에 각 리소스의 상태 정보(민감 정보 포함)가 저장됨을 확인 — 별도 보안 관리 필요 (`.gitignore`로 커밋 제외 처리)

## 4. Linux VM 접속 및 환경 점검

### 4.1 접속 정보

| 항목 | 값 |
|---|---|
| VM 이름 | bax-poc-d-vm |
| 크기 | Standard_D4s_v5 |
| 계정 | azureuser |
| 인증 방식 | 비밀번호 인증 (`disable_password_authentication = false`) |
| 네트워크 | subnet_mgmt (10.130.0.0/27), Public IP 활성화 |

> ※ 비밀번호는 `sensitive` 변수로 분리 처리되어 있으며, 실제 값은 `.gitignore` 처리된 `secrets.auto.tfvars`에서 관리함.

### 4.2 초기 환경 점검 결과 (vm-init.sh 도입 이전)

초기 구성 시 VM 접속 직후 홈 디렉터리(`~`) 점검 결과, `az` CLI 및 `kubectl` 모두 미설치 상태로 확인됨.  
이를 계기로 **VM 생성 시 자동으로 필요 도구를 설치하는 cloud-init 스크립트(`vm-init.sh`)** 를 도입하였다.

```bash
azureuser@bax-poc-d-vm:~$ az login
az: command not found
azureuser@bax-poc-d-vm:~$ kubectl
Command 'kubectl' not found ...
```

### 4.3 VM 초기화 자동 설치 스크립트 (vm-init.sh)

`Azure/scripts/vm-init.sh`를 cloud-init(`custom_data`)으로 전달하여 **VM 최초 부팅 시 자동으로** 필요 도구를 설치하도록 구성하였다.

**설치 항목**

| 도구 | 용도 |
|---|---|
| Azure CLI | `az login`, AKS 자격증명 취득 등 |
| kubectl | 쿠버네티스 클러스터 제어 |
| Helm 3 | 쿠버네티스 패키지 관리 |
| kubelogin | AKS AAD 인증 토큰 변환 |
| postgresql-client | PostgreSQL DB 접속 |
| jq / git / curl / net-tools / dnsutils | 공통 운영 유틸리티 |

**Terraform 코드 구조**

```
Azure/
├── scripts/
│   └── vm-init.sh              # DEV/PRD 공용 설치 스크립트
├── modules/linux-vm/
│   ├── variables.tf            # custom_data 변수 추가 (default: null)
│   └── main.tf                 # custom_data = var.custom_data 연결
└── environments/
    ├── BaxPOC-DEV/main.tf      # custom_data = base64encode(file(...))
    └── baxpoc-prd/main.tf      # custom_data = base64encode(file(...))
```

`environments/BaxPOC-DEV/main.tf` 적용 예시:

```hcl
module "linux-vm" {
  ...
  custom_data = base64encode(file("${path.module}/../../scripts/vm-init.sh"))
  ...
}
```

**설치 로그 확인**

VM 접속 후 아래 명령으로 설치 진행 상황 및 완료 여부를 확인한다.

```bash
cat /var/log/vm-init.log
```

정상 완료 시 마지막 줄에 아래 메시지가 출력된다.

```
=== vm-init complete: <날짜/시간> ===
```

> ⚠️ `custom_data`는 **VM 최초 생성 시에만** 실행된다. 스크립트를 변경한 경우 VM을 재생성해야 적용된다.

### 4.6 Linux VM(Jumpbox)에서 AKS 노드 접속

Linux VM(`bax-poc-d-vm`, subnet_mgmt: 10.130.0.0/27)은 AKS 노드 서브넷(`subnet_aks`: 10.130.0.128/25)과 동일 VNet에 위치하므로 Jumpbox로 활용 가능하다.  
AKS 노드에 접속하는 방법은 크게 3가지이며, 상황에 따라 선택한다.

---

#### 방법 1: kubectl debug node (권장 — SSH 키 불필요)

`kubectl debug`는 노드에 특권(privileged) 컨테이너를 띄워 호스트 파일 시스템과 프로세스에 접근하는 방식으로, SSH 키 설정 없이 가장 간편하게 노드 내부를 점검할 수 있다.

**① 노드 이름 확인**

```bash
kubectl get nodes -o wide
```

출력 예시:

```
NAME                              STATUS   ROLES   AGE   VERSION   INTERNAL-IP
aks-sysnp01-12345678-vmss000000   Ready    agent   1d    v1.29.x   10.130.0.130
aks-usernp01-12345678-vmss000000  Ready    agent   1d    v1.29.x   10.130.0.150
```

**② 노드 디버그 컨테이너 접속**

```bash
kubectl debug node/<노드이름> -it \
  --image=mcr.microsoft.com/cbl-mariner/busybox:2.0

# 예시
kubectl debug node/aks-usernp01-12345678-vmss000000 -it \
  --image=mcr.microsoft.com/cbl-mariner/busybox:2.0
```

**③ 호스트 파일 시스템 진입**

```bash
# 컨테이너 내부에서 실행
chroot /host bash
```

`chroot /host` 이후부터는 노드의 실제 OS 환경에서 명령을 실행할 수 있다.

**④ 점검 완료 후 정리**

```bash
# 컨테이너에서 나온 후 debug pod 삭제
exit
kubectl get pods -A | grep node-debugger
kubectl delete pod <debug-pod-name> -n <namespace>
```

> ⚠️ `kubectl debug node`로 생성된 Pod는 자동 삭제되지 않으므로 점검 후 반드시 수동 삭제한다.

---

#### 방법 2: SSH를 통한 직접 노드 접속

AKS 노드는 기본적으로 비밀번호 인증이 비활성화되어 있으며 SSH 공개 키 인증만 허용된다.  
Jumpbox VM과 AKS 노드가 같은 VNet에 있으므로 Private IP로 직접 SSH 접속이 가능하다.

**① AKS 노드 Private IP 확인**

```bash
# kubectl로 확인
kubectl get nodes -o wide

# 또는 Azure CLI로 확인 (MC_ 리소스 그룹 내 VMSS 인스턴스)
az vmss list-instance-public-ips \
  --resource-group <MC_리소스그룹> \
  --name <VMSS이름>
```

> AKS가 생성하는 노드 리소스 그룹 이름 형식: `MC_<RG명>_<클러스터명>_<리전>`  
> 예: `MC_bax-poc-dev-app-rg_bax-poc-dev-aks_koreacentral`

**② SSH 키 준비**

AKS 클러스터 생성 시 지정한 SSH 공개 키에 대응하는 **개인 키**가 Jumpbox에 있어야 한다.  
Terraform으로 생성한 경우 `azurerm_kubernetes_cluster`의 `linux_profile.ssh_key` 블록에 지정된 키를 사용한다.

```bash
# 개인 키 파일 권한 설정
chmod 600 ~/.ssh/id_rsa
```

**③ Jumpbox에서 AKS 노드로 SSH 접속**

```bash
ssh -i ~/.ssh/id_rsa azureuser@<노드 Private IP>

# 예시
ssh -i ~/.ssh/id_rsa azureuser@10.130.0.130
```

**④ (선택) SSH ProxyJump 설정으로 로컬 PC → Jumpbox → AKS 노드 원스텝 접속**

로컬 PC에서 Jumpbox를 경유해 AKS 노드에 한 번에 접속하려면 아래와 같이 설정한다.

```bash
# 로컬 PC에서 실행
ssh -i ~/.ssh/id_rsa \
  -J azureuser@<Jumpbox Public IP> \
  azureuser@<노드 Private IP>
```

또는 `~/.ssh/config`에 등록해 간소화:

```
Host jumpbox
  HostName <Jumpbox Public IP>
  User azureuser
  IdentityFile ~/.ssh/id_rsa

Host aks-node
  HostName <노드 Private IP>
  User azureuser
  IdentityFile ~/.ssh/id_rsa
  ProxyJump jumpbox
```

```bash
# 이후 간단히 접속
ssh aks-node
```

#### 방법 3: az aks command invoke (Private 클러스터 / kubectl 없이 실행)

`kubectl` 없이 Azure CLI만으로 AKS 클러스터에 명령을 전달할 수 있다.  
`private_cluster_enabled = true` 환경이거나 kubeconfig 설정이 어려운 경우에 유용하다.

```bash
az aks command invoke \
  --resource-group bax-poc-dev-app-rg \
  --name bax-poc-dev-aks \
  --command "kubectl get nodes -o wide"
```

파일을 함께 전달해야 하는 경우:

```bash
az aks command invoke \
  --resource-group bax-poc-dev-app-rg \
  --name bax-poc-dev-aks \
  --command "kubectl apply -f deployment.yaml" \
  --file deployment.yaml
```

| 방법 | SSH 키 필요 | 노드 OS 접근 | 권장 상황 |

## 5. 후속 조치 권고 사항

- **[검증]** VM에서 각 서비스 Private Endpoint DNS 정상 해석 여부 확인 (`nslookup`), AKS 노드 상태 (`kubectl get nodes`), PostgreSQL 접속 테스트
- **[AI 모델]** AI Foundry 계정은 생성 완료 상태이며, 실제 모델 배포(`azurerm_cognitive_deployment` 또는 포털)는 별도 작업으로 진행 필요
- **[AKS 연동]** ACR(`baxpocdcr`)과 AKS 연동 (`az aks update --attach-acr`) 후 애플리케이션 이미지 빌드/배포
- **[DB 초기화]** PostgreSQL에 POC용 데이터베이스/스키마 생성
- **[보안 정리]** `admin_password` / `administrator_password`를 Key Vault 또는 `sensitive` 변수로 분리, ACR/AKS의 퍼블릭 액세스 설정 재검토 (현재는 `public_network_access_enabled = true`, `private_cluster_enabled = false`)
- **[모니터링]** kubernetes 모듈의 `log_analytics_workspace_id` 변수에 워크스페이스를 연결하여 AKS 모니터링 구성
